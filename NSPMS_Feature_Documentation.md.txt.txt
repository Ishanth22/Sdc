## 1. Authentication & Access Control

### What It Is
A secure, role-based authentication system that controls who can access what. There are three user roles — **Founder**, **Investor**, and **Admin** — each with a different set of pages and permissions.

### How It Works

#### Login
- User enters email + password on the Login page (`/login`).
- The frontend calls `POST /api/auth/login`.
- The backend fetches the user from MongoDB, compares the submitted password against the stored **bcrypt hash**.
- **Account Locking:** After **5 failed login attempts**, the account is locked for **15 minutes**. The lock auto-lifts after the timer expires.
- On success, a **JWT token** (JSON Web Token) is returned and stored in the browser's `localStorage`.
- All subsequent API calls carry this token in the `Authorization: Bearer <token>` header.

#### Registration
- New founders register from the same Login page (toggled to "Register" mode).
- The system creates a User record + a blank StartupProfile linked to that user.

#### Role-Based Routing
- The frontend uses a [ProtectedRoute](file:///c:/Users/Mohammed%20Muksith%20S/sdc/nspms/frontend/src/components/ProtectedRoute.tsx#10-29) component that checks `user.role` before allowing access.
- **Founder** → sees Dashboard, Metrics, Forecasting, AI Risk, AI Advisor, Milestones, KPIs, Reports, Simulation.
- **Investor** → sees the Investor Dashboard (read-only view of all startups).
- **Admin** → sees the Admin Dashboard (manage all startups, users, subscriptions).

#### Demo Credentials
| Email | Password | Role |
|---|---|---|
| founder1@nspms.in | password123 | Founder |
| investor@nspms.in | password123 | Investor |
| admin@nspms.in | password123 | Admin |

#### Implementation Type
> 🔧 **Rule-Based / Standard Auth** — No AI involved. Uses industry-standard JWT + bcrypt.

### Fields Captured
| Field | Description |
|---|---|
| Company Name | Official name of the startup |
| Sector | Industry (Fintech, SaaS, Healthtech, Edtech, etc.) |

#### Implementation Type

## 3. Metrics Submission

### What It Is
The primary data entry point for founders. Each submission captures one month's performance across four categories: Financial, Operational, Innovation, and Impact.

### Fields Captured

**Financial**
- Monthly Revenue (₹)
- Monthly Expenses (₹)
- Burn Rate (₹/month)
- Runway (months of cash remaining)
- Total Funding Raised (₹)
- Latest Funding Amount, Type (Equity/Debt/Grant), and Investor Name

**Operational**
- Active Users (monthly)
- New User Signups
- Customer Acquisition Cost (CAC) (₹)
- Lifetime Value (LTV) (₹)
- Churn Rate (%)
- Gross Merchandise Value (GMV) (₹)
- Cities Served

**Innovation**
- Patents Filed / Granted
- Trademarks Filed
- R&D Spend (₹)

**Impact**
- Direct Jobs Created
- Women Employees
- Rural Employees
- Exports (₹)

### What Happens After Submission
When a founder submits metrics, the backend automatically triggers:
1. **Vitality Score calculation** — computes a new health score for that period.
2. **Alert generation** — checks all thresholds and creates warnings if needed.
3. **Cache invalidation** — wipes all AI forecasting, risk, and benchmark caches so the next page visit regenerates with fresh AI analysis.

#### Implementation Type
> 🔧 **Standard CRUD + Rule-Based Triggers** — No AI in the submission itself. AI is triggered on the next visit to forecasting/risk pages.

## 5. Startup Vitality Score (Health Score)

### What It Is
A single composite score (0–100) that summarizes the startup's overall health. Calculated automatically every time new metrics are submitted.

### How the Score is Calculated (Rule-Based Formula)

Each of the 5 components is scored independently (0–100), then weighted:

```
Final Score = (revenueGrowth × 0.25) + (userGrowth × 0.20) + 
              (burnEfficiency × 0.20) + (churnStability × 0.15) + 
              (runwayStability × 0.20)
```

**Revenue Growth Score**
- Compares current month vs previous month revenue.
- 100 if growth ≥ 30%, scales down to 0 if declining.

**User Growth Score**
- Compares active user count month-over-month.
- Scaled against sector typical growth rates.

**Burn Efficiency Score**
- Ratio of revenue to burn rate. Score = 100 if revenue ≥ burn (profitable). Drops sharply when burn > revenue.

**Churn Stability Score**
- Score = 100 if churn = 0%. Drops as churn rises. Score = 0 if churn ≥ 20%.

**Runway Stability Score**
- Score = 100 if runway ≥ 18 months. Drops linearly. Score = 0 if runway < 1 month.

### Derived Outputs (also rule-based)
- **Risk Level:** Low (0 flags), Moderate (1–2 flags), High (3 flags), Critical (4+ flags)
- **Risk Flags:** Auto-generated text like "Low runway," "High churn," "Falling revenue"
- **Funding Readiness %:** Points given for: revenue > 0, revenue > ₹5L, users > 1000, team ≥ 20, runway ≥ 6 months, health score ≥ 60
- **Score Explanation:** Auto-generated bullet points explaining what is helping or hurting the score.

#### Implementation Type
> 🔧 **Fully Rule-Based (Mathematical)** — No AI. Purely weighted formula calculated in [vitalityScore.ts](file:///c:/Users/Mohammed%20Muksith%20S/sdc/nspms/backend/src/services/vitalityScore.ts).

## 6. Smart Alerts System

### What It Is
An automated notification system that watches the startup's metrics and generates actionable alerts when something needs attention.

### Alert Types and Thresholds

| Alert | Trigger | Severity |
|---|---|---|
| Critically Low Runway | Runway < 3 months | Critical 🚨 |
| Low Runway Warning | Runway < 6 months | Warning ⚠️ |
| Revenue Decline | Revenue dropped > 20% month-over-month | Warning ⚠️ |
| High Churn | Churn rate > 15% | Critical 🚨 |
| Elevated Churn | Churn rate > 8% | Warning ⚠️ |
| LTV/CAC Danger | LTV/CAC ratio < 1x | Critical 🚨 |
| Unhealthy LTV/CAC | LTV/CAC ratio < 3x | Warning ⚠️ |
| High Burn Relative to Revenue | Burn > 3× revenue | Warning ⚠️ |

### How It Works
- Alerts are generated server-side when `POST /api/metrics` is called.
- Stored in the [Alert](file:///c:/Users/Mohammed%20Muksith%20S/sdc/nspms/frontend/src/pages/Dashboard.tsx#37-43) MongoDB collection.
- Shown on the Dashboard with dismiss functionality.
- Old alerts for the same period are cleared before new ones are created (deduplication).

#### Implementation Type
> 🔧 **Rule-Based** — Pure threshold comparisons. No AI.

## 7. AI-Powered Forecasting

### What It Is
A 6-month forward-looking projection of revenue, expenses, user growth, and cash runway. The charts show historical data (solid line) connected to AI-predicted future values (dashed line).

### The Forecasting Flow (Step by Step)

```
1. Founder visits /forecasting
2. Backend checks MongoDB cache
   ├── CACHE HIT → Return cached AI result instantly (no API call)
   └── CACHE MISS → Proceed to generate
3. Rule-Based Baseline is computed (math fallback)
4. AI is called with full startup context (no baseline numbers shared)
5. AI generates 6-month forecast independently
   ├── SUCCESS → Cache AI result → Return to frontend
   └── FAILURE / TIMEOUT → Fall back to rule-based result
```

### What the AI Receives (Prompt Context)
- Startup name, sector, stage, city, team size
- Sector average monthly growth rate
- Seasonal boost months and percentages
- Regulatory burden level (low/medium/high)
- Full historical metrics — all submitted months with revenue, expenses, burn, runway, users, CAC, LTV, churn

### What the AI Generates
The AI outputs structured JSON with:
- **6 months of revenue forecasts** (e.g., April 2025: ₹7,20,000)
- **6 months of expense forecasts**
- **6 months of user growth forecasts**
- **Projected runway runout date**
- **Confidence level** (low / medium / high)
- **5 data-driven insights** specific to the startup
- **Overall strategic outlook** (2–3 sentences)
- **Critical action** — the single most important thing to do in the next 30 days

### The Charts

| Chart | X-Axis | Y-Axis | Lines |
|---|---|---|---|
| Revenue Forecast | Period (YYYY-MM) | ₹ Amount | Solid = actual, Dashed = AI forecast |
| Expense Forecast | Period (YYYY-MM) | ₹ Amount | Solid = actual, Dashed = AI forecast |
| User Growth Forecast | Period (YYYY-MM) | User Count | Solid = actual, Dashed = AI forecast |
| Runway Projection | Period (YYYY-MM) | Months of runway | Computed from AI's revenue/expense curves |

### Smart Caching
- Results are cached in MongoDB with a **data hash** (SHA-256 of model name + all metric values + sector + stage).
- Cache is automatically invalidated when:
  - New metrics are submitted (any change in data).
  - The AI model is changed (model name is part of the hash).
- Cache is persistent across server restarts (stored in DB, not RAM).
- The frontend shows: 🤖 **AI-Enhanced** badge (purple, pulsing) when AI-powered, and the exact cache timestamp.

### Fallback
If the AI API fails or times out (90-second timeout), the system silently falls back to the **rule-based forecast** (weighted moving average + linear regression). The badge will not show as "AI-Enhanced."

#### Implementation Type
> 🤖 **Primarily AI-Powered** (model: `arcee-ai/trinity-large-preview:free` via OpenRouter)  
> 🔧 **Rule-Based Fallback** when AI is unavailable.

## AI Architecture Summary

```
OpenRouter API (arcee-ai/trinity-large-preview:free)
         │
         ├── forecasting.ts     → 6-month revenue/expense/user forecasts
         │                         Temperature: 0.6, Max tokens: 4096
         │                         Timeout: 90 seconds
         │
         ├── aiRiskEngine.ts    → Risk prediction + benchmarking analysis
         │                         Called on /ai-risk page visit
         │
         └── aiAdvisor.ts       → Conversational chatbot answers
                                   Called on every user message

MongoDB Cache (AnalysisCache collection)
         ├── cacheType: 'forecast'         → Forecasting results
         ├── cacheType: 'risk-prediction'  → Risk analysis results
         └── cacheType: 'benchmark'        → Benchmarking results

Cache Invalidation Trigger:
  POST /api/metrics → invalidateAllCaches() → deletes all 3 caches
  → Next page visit regenerates all AI results fresh

Cache Key (dataHash):
  SHA-256( modelName + allMetrics + sector + stage )
  Changing the model OR submitting new metrics = cache miss = new AI call
```
